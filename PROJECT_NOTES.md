# 북한산 논문 아카이브 — 프로젝트 진행 요약

다른 AI 툴(ChatGPT, Gemini 등)에 이 파일을 통째로 붙여넣고 "이 프로젝트 이어서 도와줘"라고 하면 맥락을 파악할 수 있게 정리한 문서입니다.

## 목표
북한산 관련 학술논문을 자동으로 검색·수집해서 카테고리별로 분류해 웹사이트로 보여준다.
GitHub Pages(정적 사이트) + GitHub Actions(배치 방식 자동 수집) 구조.

## 저장소 구조
```
index.html                        ← 실제 사이트 (GitHub Pages로 서빙)
fetch_naver.py                    ← 논문 수집 스크립트 (GitHub Actions가 배치로 실행)
papers.json                       ← 수집 결과 (누적됨, index.html이 읽음)
fetch_state.json                  ← 어느 배치까지 처리했는지 기억하는 파일
.github/workflows/fetch-naver.yml ← 매일 배치 1개씩 자동 실행하는 워크플로우
README.md                         ← 설정 가이드
```

## 지금까지의 시행착오 (중요 — 같은 실수 반복하지 말 것)

1. **RISS 스크래핑 실패**: GitHub Actions의 IP를 RISS가 봇으로 인식해서 항상 빈 페이지를 반환. Playwright, requests+BeautifulSoup 다 시도했지만 방식 문제가 아니라 IP 차단 문제였음 → RISS 포기.

2. **네이버 openapi "academic" API는 애초에 존재하지 않음**: `openapi.naver.com/v1/search/academic.json`은 없는 엔드포인트. 논문류를 다루는 진짜 API는 `doc.json`("전문자료 검색")인데, 이것도 title/link/description 3개 필드뿐이라 저자·학술지·초록 정보가 없음 → API 자체를 포기.

3. **현재 방식: academic.naver.com 웹페이지를 직접 스크래핑**. 실제 HTML 구조를 사용자가 브라우저에서 직접 복사해서 확인함:
   - 검색결과 목록: `https://academic.naver.com/search.naver?field=0&docType=1&query=검색어`
     - 결과 블록: `div.ui_listing_info`
     - 제목: `h4 a.ui_listing_subtit` (href가 상세페이지 링크: `/article.naver?doc_id=...`)
     - 무료/유료 배지: `h4 span.spimg` (텍스트 "무료"/"유료")
     - 연도/저자/학술지: `div.ui_listing_desc` 안의 `span.ui_listing_source` 여러 개 (연도는 4자리 숫자로 구분, 학술지는 내부에 `<a>` 태그 있음, 나머지는 저자)
     - 피인용수: `span.ui_listing_cited_num`
     - 요약 스니펫: `p.ui_listing_txt` (주의: 이건 영어 제목 번역 + 초록 일부가 섞인 짧은 스니펫이라 진짜 초록이 아님)
   - 상세 페이지: `https://academic.naver.com/article.naver?doc_id={doc_id}`
     - 진짜 초록: `div#div_abstract p.ui_enddetail_txt` (한글 초록 + `<br><br>` + 영어 초록이 함께 들어있음)
     - 소속(institution) 정보는 이 페이지에서 확인 안 됨(대부분 빈 값)

4. **GitHub Actions에서 스크래핑하다가 완전히 막힌 적이 있었음**: 24개 국립공원으로 스코프를 넓혔더니 요청이 급증했고, 그 이후로는 가장 단순한 요청 하나까지 100% 타임아웃 나는 상태가 됨. 처음엔 "IP 자체가 차단됐다"고 결론 내리고 로컬 실행으로 전환했었으나, 사용자가 파이썬이 없어서 **다시 GitHub Actions로 되돌리되, 이번엔 "짧은 시간에 요청이 몰려서 서버가 의도적으로 응답을 늦춘 것"이라는 가설로 대응함**:
   - 요청마다 2~5초 랜덤 대기 추가 (`polite_sleep()`)
   - 검색어 17개를 한 번에 다 처리하지 않고 **배치로 나눠서(한 번에 3개씩) 처리**하고, `fetch_state.json`에 진행 상황을 저장해서 다음 실행 때 이어서 처리
   - `papers.json`은 매 실행마다 덮어쓰지 않고 기존 데이터에 누적(merge)
   - 워크플로우 스케줄을 매주 1회 → **매일 1회**로 변경 (배치라 한 번에 처리하는 양이 적어서 매일 돌려도 부담이 적음)
   - **이 방식도 통하지 않으면(계속 타임아웃 나면) 결국 IP 차단이 맞다는 뜻이므로, 그때는 다시 로컬 실행으로 돌아가야 함.**

5. **AI(Gemini) 연동 시도했다가 걷어냄**:
   - `gemini-1.5-flash` 모델은 서비스 종료됨(2025년 4월부터 신규 프로젝트 사용 불가) → `gemini-2.5-flash-lite`로 교체했었음.
   - 코드 버그: AI가 반환한 category가 5개 라벨과 안 맞으면 summary/usage까지 통째로 버리던 문제가 있었음 → 수정했었음.
   - 이 두 수정이 실제로 잘 작동하는지 깨끗하게 확인되기 전에 24개 공원 확장 + 차단 문제가 겹쳐서 혼란스러워짐 → **AI 기능은 완전히 제거하고 규칙기반 분류로 되돌림.** 필요하면 스크래핑이 완전히 안정된 뒤에 다시 검토.

6. **카카오맵도 제거함**: 스코프를 북한산 단독으로 단순화하면서 지도 기능도 같이 걷어냄 (필요하면 나중에 재도입 가능).

## 현재 상태 (이 문서 작성 시점)
- **북한산만, AI 없이, 지도 없이, 규칙기반 분류로만** 동작하는 버전.
- **GitHub Actions에서 배치 방식으로 실행** — 한 번 실행에 검색어 3개씩만 처리하고, `fetch_state.json`으로 진행 상황을 기억해서 다음 실행 때 이어감. 검색어 17개를 한 바퀴 다 돌려면 총 6번 실행 필요. 매일 자동 실행되도록 스케줄 설정됨.
- 유지된 기능: 논문 제목+초록에서 자주 나오는 한글 키워드를 뽑아 `papers.json`의 `keywords` 필드에 저장하고, 사이트에 클릭 가능한 태그 클라우드로 표시.
- academic.naver.com 요청에는 재시도(최대 3회)와 브라우저 유사 헤더, 요청마다 2~5초 랜덤 대기를 적용해둔 상태.

## 사이트 구조 (index.html)
- 좌측 패널: 키워드 태그 클라우드 + 검색창 + 카테고리 필터(자원조사/재난/탐방/생태/역사문화) + 논문 목록
- 우측 패널: 목록에서 논문 클릭 시 상세정보(저자/소속/학술지정보/발행정보/초록) 표시
- 지도, AI 관련 UI 없음
