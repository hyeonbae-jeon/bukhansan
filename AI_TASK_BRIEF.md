# 북한산 논문 아카이브 — AI 작업 지시서

이 문서는 다른 AI(ChatGPT, Gemini, 다른 Claude 세션 등)에게 그대로 붙여넣어 "이 요구사항대로 사이트를 만들어줘/이어서 작업해줘"라고 지시할 수 있도록 작성한 명세서입니다.

---

## 프로젝트 목표

북한산 관련 학술논문을 자동으로 검색·수집해서 카테고리별로 분류하고, 한눈에 훑어보고 상세 내용까지 확인할 수 있는 웹사이트를 만든다. 사람이 수동으로 논문을 검색할 필요 없이, 주기적으로 자동 수집되고 사이트에 반영되어야 한다.

## 전체 아키텍처

- **정적 사이트(GitHub Pages) + GitHub Actions에서 배치 방식으로 데이터 수집**.
- Python 스크립트(`fetch_naver.py`)가 GitHub Actions에서 실행되어 논문을 검색·수집해서 `papers.json`에 저장하고 저장소에 자동 커밋한다.
- **한 번 실행에 검색어를 전부 처리하지 않고 일부만(배치) 처리**하고, `fetch_state.json`에 진행 상황을 기록해서 다음 실행 때 이어간다. `papers.json`은 매번 덮어쓰지 않고 기존 데이터에 새로 찾은 것만 누적한다.
- 매일 자동 실행 + 수동 실행 버튼 둘 다 지원한다 (GitHub Actions `schedule` + `workflow_dispatch`).
- 웹사이트(`index.html`)는 서버 없이 `papers.json`을 fetch해서 화면에 그린다.

## 데이터 수집 요구사항

### 어디서 데이터를 가져와야 하는가
**`academic.naver.com`(네이버 학술정보) 웹페이지를 직접 읽어온다(스크래핑).** 아래 이유로 다른 방법은 이미 실패했다:
- RISS는 GitHub Actions의 IP를 봇으로 인식해 항상 빈 페이지를 반환한다 (스크래핑 방식을 바꿔도 소용없음, IP 차단 문제).
- `openapi.naver.com`의 "academic" 검색 API는 **존재하지 않는다.** 진짜 있는 API(`doc.json`, "전문자료 검색")는 title/link/description 3개 필드뿐이라 저자·학술지·초록 정보를 못 얻는다.
- 따라서 `academic.naver.com` 웹페이지 자체를 스크래핑하는 게 유일하게 필요한 데이터를 다 얻을 수 있는 방법이다.

### 실제 확인된 HTML 구조 (반드시 이 selector를 써야 함 — 추측하지 말 것)
검색결과 목록 페이지: `https://academic.naver.com/search.naver?field=0&docType=1&query={검색어}&start={시작번호}`
- 결과 블록: `div.ui_listing_info`
- 제목: `h4 a.ui_listing_subtit` (href = 상세페이지 링크 `/article.naver?doc_id=...`)
- 무료/유료 배지: `h4 span.spimg` (텍스트 "무료" 또는 "유료")
- 연도/저자/학술지: `div.ui_listing_desc` 안의 `span.ui_listing_source` 여러 개
  - 4자리 연도 숫자면 연도
  - 내부에 `<a>` 태그가 있으면 학술지명
  - 나머지는 저자명
- 피인용수: `span.ui_listing_cited_num`
- 짧은 설명 스니펫: `p.ui_listing_txt` — **주의: 이건 영어 제목 번역 + 초록 일부가 섞인 스니펫이라 진짜 초록이 아니다. 목록 미리보기용으로만 쓰고, 진짜 초록은 아래 상세페이지에서 따로 가져와야 한다.**

상세 페이지: `https://academic.naver.com/article.naver?doc_id={doc_id}`
- 진짜 초록: `div#div_abstract p.ui_enddetail_txt` (한글 초록 + `<br><br>` + 영어 초록이 함께 들어있음. 한글 부분만 쓰고 싶으면 첫 문단만 발췌)
- 소속(institution) 정보는 이 페이지에서 거의 확인 안 됨 (없으면 그냥 빈 값 처리)

### 스크래핑 시 주의할 것 (매우 중요)
- 예전에 GitHub Actions로 요청이 급증(24개 국립공원으로 스코프 확장)했다가 이후 모든 요청이 100% 타임아웃 나는 상태가 된 전례가 있다. 처음엔 "IP 자체가 차단됐다"고 보고 로컬 실행으로 바꿨었지만, 사용자가 로컬에 Python이 없어서 다시 GitHub Actions로 복귀하면서 **"짧은 시간에 요청이 몰려서 서버가 응답을 늦춘 것"이라는 가설로 아래처럼 대응 중**:
  - 요청마다 2~5초 랜덤 대기
  - 검색어 전체를 한 번에 처리하지 않고 **배치로 나눠서(3개씩) 처리**, 진행 상황을 `fetch_state.json`에 저장해 다음 실행 때 이어감
  - `papers.json`은 실행마다 덮어쓰지 않고 누적
  - 스케줄은 매일 1회(배치라 부담이 적어서)
- **이렇게 해도 계속 타임아웃 나면, 결국 IP 차단이 맞다는 뜻이다.** 그때는 GitHub Actions를 포기하고 로컬 실행(사용자 컴퓨터에서 직접 `python3 fetch_naver.py` 실행 후 git push)으로 다시 전환해야 한다. 다만 로컬 실행은 사용자가 Python 설치 등 진입장벽이 있어서, 가능하면 배치+속도조절 방식으로 GitHub Actions를 계속 쓰는 걸 우선 시도할 것.

### 검색 범위
- "북한산" 하나만으로 검색하면 결과가 적으므로, 여러 변형 검색어(예: "북한산", "북한산국립공원", "북한산 생태", "북한산 재난", "북한산 탐방", "북한산 역사", "북한산성", "북한산 등산로", "북한산 산림", "북한산 문화재", "북한산 사찰", "북한산 산사태", "북한산 식생", "북한산 둘레길", "북한산 지질", "북한산 토양", "북한산 관리" 등)을 던져서 모은다.
- 검색결과는 페이지네이션(`start` 파라미터, 한 페이지 20건)을 끝까지 따라가서 **가능한 한 많이(100건 이상)** 모은다. 새로운 결과가 더 안 나오면(중복만 반복되면) 그 검색어는 그만 돈다.
- 제목이나 스니펫에 "북한산"이 아예 없는 결과는 무관한 결과로 보고 제외한다.
- 같은 논문이 여러 검색어에서 중복으로 나올 수 있으므로 제목 기준으로 중복 제거한다.

## 사이트에 표시해야 할 정보

### 분류 (카테고리)
아래 5개 카테고리로 분류하고, 색깔 있는 배지로 구분해서 보여준다:
- **자원조사** (기본값, 아래 항목에 해당 안 되는 것)
- **재난** (산사태, 산불, 침수, 안전사고 등)
- **탐방** (등산객·탐방객 행태, 이용 실태, 관광)
- **생태** (동식물, 식생, 서식지, 산림, 생물다양성)
- **역사문화** (역사, 문화재, 유적, 사찰)

분류는 제목+초록 내용을 보고 판단해야 하며, 검색어만 보고 기계적으로 분류하면 안 된다(예: "북한산 생태"로 검색했다고 무조건 생태로 분류하면 안 됨 — 실제로는 재난 논문이 섞여 나올 수 있음).

### 목록(카드) 화면에 보여줄 것
- 카테고리 배지
- 논문 제목 (잘리지 않고 전체 표시)
- 연도 | 저자 | 학술지 (구분자로 표시, 정보 없는 항목은 생략)
- 초록 앞부분 발췌(150자 정도) — **영어 제목 번역이 아니라 진짜 초록의 한글 부분**을 보여줄 것
- 원문 보기 링크

### 클릭 시 상세정보에 보여줄 것 (네이버 학술정보 자체 상세페이지 형식과 동일하게)
- 제목
- 저자
- 소속
- 학술지정보
- 발행정보 (학술지명 · 연도 · 피인용수)
- 초록 (전체)
- 원문 페이지 링크

(지도 기능은 현재 빠져있는 상태 — 아래 "보류한 것" 참고)

### 부가 기능
- 카테고리 필터 버튼
- 텍스트 검색 (제목/초록 내용 대상)
- **자주 나오는 키워드 태그 클라우드**: 수집된 논문들의 제목+초록에서 자주 등장하는 한글 단어를 뽑아 빈도순으로 클릭 가능한 태그로 보여준다. 클릭하면 그 단어로 검색창이 채워지고 목록이 필터링된다. (형태소 분석기 없이 정규식+조사 제거 휴리스틱으로 구현되어 있어 100% 정확하진 않다는 점 감안.)

## 화면 레이아웃
- 상단: 사이트 제목 + 현재 수집 건수/마지막 업데이트 시각
- 좌측 패널: 검색창 → 카테고리 필터 → 키워드 태그 클라우드 → 논문 목록(카드 나열)
- 우측 패널: 목록에서 논문을 클릭하면 그 논문의 상세정보가 뜬다 (별도 팝업이 아니라 화면 우측에 고정된 패널 형태로).

## 기술 스택
- 백엔드 없음. `fetch_naver.py`(Python, requests + BeautifulSoup)가 **GitHub Actions에서 배치 방식으로 실행**되어 `papers.json`을 갱신하고 저장소에 자동 커밋.
- 프론트엔드는 순수 HTML/CSS/JavaScript 한 파일(`index.html`), 프레임워크 없음. `papers.json`을 fetch로 읽어서 렌더링.
- 지도(카카오맵)는 현재 빠져있음.
- GitHub Pages로 정적 호스팅.

## 지금까지 시도했다가 보류한 것 (재도입 검토 시 참고)
- **카카오맵 연동**: 논문 위치를 지도로 보여주는 기능이 있었으나 스코프를 단순화하면서 제거함. 재도입 시 카카오 개발자센터에서 JavaScript 키 발급 필요.
- **AI(Gemini) 연동으로 분류/요약/활용방안 자동 생성**: 시도했으나 (1) 24개 국립공원으로 스코프를 넓히자 스크래핑 자체가 불안정해졌고 (2) AI 응답 파싱 로직에 버그도 있었어서 일단 걷어냄. 다시 시도한다면 스크래핑 부분을 먼저 완전히 안정화한 뒤에 AI 기능을 별도로 얹는 순서를 권장. (참고: `gemini-1.5-flash` 모델은 서비스 종료됐으니 최신 모델명 확인 후 사용할 것.)
- **전국 24개 국립공원으로 확장**: 국립공원 24개소(2023년 팔공산, 2026년 금정산 신규 지정분 포함) 명단과 대략적 중심 좌표는 이전 대화에 정리되어 있음. 스코프를 넓힐 땐 한 번에 다 하지 말고, 로컬 실행 기준으로도 요청 속도를 충분히 낮추고 공원 몇 개씩 나눠서(예: 하루에 2~3개씩) 여러 날에 걸쳐 점진적으로 시도할 것 — 한꺼번에 하면 이번엔 사용자 본인의 IP가 차단될 위험이 있음.

## 현재 상태
북한산 단독, AI 없이 규칙기반 분류만 사용, 지도 없이 순수 목록+상세정보만 있는 버전. 논문 수집은 **GitHub Actions에서 배치 방식(검색어 3개씩, 매일 자동 실행)**으로 진행되며 `papers.json`에 누적됨. 키워드 태그 클라우드 기능은 유지됨. 이 배치 방식으로도 계속 타임아웃이 나는지, 아니면 정상적으로 데이터가 쌓이는지 확인하는 게 다음 우선순위 — 계속 안 되면 결국 로컬 실행으로 다시 돌아가야 할 수 있음.
